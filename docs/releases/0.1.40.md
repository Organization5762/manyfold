# manyfold 0.1.40

manyfold 0.1.40 expands the Rust-backed PubSub stream runtime into a
namespace-scoped worker substrate for Heart-style data processing.

## Features

- Adds `PubSubTopic(...)` for namespace-scoped topic handles that boot or reuse
  one PubSub runtime per namespace.
- Moves `ManyFoldLock` into the Rust/PyO3 layer with process-local coordination,
  cross-process file locking on the local node, and release-on-process-death
  behavior from OS file locks.
- Adds current-process/thread worker scheduling metadata and default
  ServiceDiscovery dependency metadata for PubSub-created workers.
- Adds SQL helpers for common stream operations, including filtering,
  projection, `take`, `distinct_until_changed`, `pairwise`, `latest_join`, and
  recursive numeric accumulation.
- Adds live `PubSubObservable` operators for callback-oriented pipelines,
  including `start_with`, `scan`, `with_latest_from`, `do_action`, `callback`,
  `flat_map`, `pipe`, and `switch_latest`.

## Runtime Notes

- SQL query execution now releases the Python GIL while waiting on the native
  stream-processor lock and while SQLite prepares and steps rows. Python row
  materialization and Python callbacks still run with the GIL.
- The SQL projection table is bounded by the PubSub `retained_messages` window
  and includes indexes for offset pruning, scoped offset joins, and scoped
  event-time ordering.
- PubSub still does not provide full callback subscription or stream operators
  at the ManyFold core API layer, so Heart IO/stream replacement remains an
  adapter-level bridge rather than a complete core PubSub substitution.
