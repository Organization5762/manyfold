# manyfold 0.1.29

Release date: 2026-05-09

## Summary

manyfold 0.1.29 is a backward-compatible patch release that defines validation
at the native PyO3 boundary. Native reference constructors and native write
paths now reject ambiguous identifiers and boolean epoch-like values instead of
silently coercing them into route metadata or scheduler state.

## What Changed

### Defined native reference validation

- `NamespaceRef`, `SchemaRef`, `RouteRef`, `ProducerRef`, `RuntimeRef`,
  `ClockDomainRef`, `PayloadRef`, `TaintMark`, and `ControlLoop` now reject
  blank textual identifiers with `ValueError`.
- Native `SchemaRef` versions must be positive integers, matching the Python
  `Schema` contract.
- Native `PayloadRef.logical_length_bytes` and `ScheduleGuard.not_before_epoch`
  now reject booleans instead of treating them as `0` or `1`.

### Matched native control epochs to the Python facade

- Native `Graph.emit(..., control_epoch=...)` and
  `WritablePort.write(..., control_epoch=...)` now reject booleans, negative
  values, and non-integers with
  `ValueError("control_epoch must be a non-negative integer or None")`.
- Failed native epoch validation happens before any envelope is written, so bad
  calls no longer leave a latest value behind.

## Compatibility

This is a patch release.

- No public APIs were removed.
- Existing valid native identifiers, positive schema versions, byte lengths,
  schedule epochs, and control epochs keep the same behavior.
- Code that passed blank identifiers, zero schema versions, booleans, negative
  epochs, or non-integer native epochs now fails fast with a stable error.

## Versioning Decision

This release is tagged as 0.1.29 because native-boundary validation is
user-visible behavior hardening. It keeps valid runtime calls working while
turning previously implicit Python-to-Rust coercions into defined errors.
