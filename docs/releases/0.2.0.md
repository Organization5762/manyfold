# manyfold 0.2.0

Release date: 2026-05-07

## Summary

manyfold 0.2.0 is a minor release that turns loose local-runtime configuration
coercion into defined validation. Sensor IO helpers now reject ambiguous boolean,
fractional, string, and invalid retry inputs at construction time instead of
letting them fail later inside queues, range loops, backoff math, or graph
capacitors.

## What Changed

### Defined local runtime control values

- Retry policies now require a positive integer `max_attempts` and validate that
  `retry_on` contains exception classes before a retry loop runs.
- Backoff delay lookup now requires a positive integer attempt index.
- Manual clocks and timing policies now require real numeric values, rejecting
  booleans and numeric-looking strings instead of silently coercing them.
- Bounded buffers, sequence counters, debug taps, double buffers, frame
  assemblers, and rate-matched sensors now reject non-integer count and capacity
  values immediately.
- Mutable debug-tap `history_size` changes are revalidated before rebuilding
  the retained history deque.

## Compatibility

This is a minor release.

- No public APIs were removed.
- Valid integer and timing values keep their existing behavior.
- Invalid values that previously relied on Python coercion, such as `True`,
  `"2"`, or `1.5` for integer controls, now raise `ValueError` with a named
  field.
- `SensorRetryPolicy(retry_on=[ValueError])` is accepted and normalized to a
  tuple, while non-exception retry entries are rejected.

## Versioning Decision

This release is tagged as 0.2.0 because it deliberately tightens public
configuration semantics for local sensor runtime controls. The change is small
and source-compatible for valid inputs, but invalid coercion paths now fail
earlier with defined `ValueError` diagnostics.
