# manyfold 0.1.7

Release date: 2026-05-03

## Summary

manyfold 0.1.7 is a backward-compatible patch release for durable route memory.
It prevents `Memory.resume()` followed by `Memory.remember()` from appending the
same resumed latest value back into the JSONL memory file.

## What Changed

### Fixed resume-to-record duplicate writes

`Memory.resume()` now tracks the graph events it republishes as already seen by
the same `Memory` instance. If callers resume a graph and then attach
`remember()` with default latest replay, the replayed latest value is suppressed
instead of being persisted a second time.

Before this release, a common restore pattern could duplicate the newest
persisted record:

```python
memory.resume(graph, route)
subscription = memory.remember(graph, route)
```

The resumed event was written into the graph, and `remember()` immediately saw
that latest event as fresh. The durable file could then grow duplicate records
without any new source observation.

### Added regression coverage

The component tests now cover resuming a durable memory file and then attaching a
recorder with latest replay enabled.

## Why This Matters

Durable route memory is intended to restore state across process restarts. A
restart should not mutate the memory file unless new observations arrive.
Avoiding restore-time duplicates keeps replay deterministic and prevents memory
logs from growing on every process boot.

## Compatibility

This is a patch release.

- No public APIs were removed or renamed.
- No call signatures changed.
- New observations after a graph restart are still recordable.
- Only values republished by the same `Memory.resume()` call are marked as seen.

## Versioning Decision

This release is tagged as `0.1.7` rather than `0.2.0` because it fixes duplicate
durable-memory writes in an existing public API without changing the intended
API shape.

## Suggested GitHub Release Notes

### Highlights

- Fixed `Memory.resume()` followed by `Memory.remember()` duplicating resumed
  latest values.
- Kept restart recording semantics for genuinely new observations unchanged.
- Added regression coverage for the restore-and-record workflow.
- Bumped package version to `0.1.7`.

### Detailed Notes

manyfold `0.1.7` fixes a durable memory replay issue where attaching a recorder
after restoring a graph could append the restored latest value back to the same
memory file. `Memory.resume()` now marks its own republished events as seen, so
restore flows stay idempotent while new observations remain recordable.
