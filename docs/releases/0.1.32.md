# manyfold 0.1.32

Release date: 2026-05-12

## Summary

manyfold 0.1.32 is a backward-compatible patch release that defines local
sensor lifecycle timing behavior that was previously delegated to Python's
threading primitives. Stop-token waits, run-loop joins, graph-node joins, and
detection-node joins/disposals now reject ambiguous timeout values before they
reach runtime thread operations.

This release also packages the post-0.1.31 main-branch hardening work across
sensor filters, frame assembly, primitive enum validation, component schemas,
component record containers, route retention policies, graph diagnostic labels,
memory/route inputs, read-write step construction, embedded route rules, example
catalog metadata, TOML-backed metadata tests, and reactive sequence snapshots.

## What Changed

### Defined lifecycle timeout validation

- `StopToken.wait()` now accepts only `None` or non-negative finite numeric
  timeout values.
- `ManagedRunLoopHandle.join()` / `.dispose()`,
  `ManagedGraphNodeHandle.join()` / `.dispose()`, and
  `DetectionNodeHandle.join()` / `.dispose()` now apply the same timeout rule.
- Boolean, string, object, non-finite, and negative timeout values now raise
  stable `ValueError` diagnostics instead of relying on platform-specific
  `threading` behavior.

### Hardened run-loop startup and disposal

- `ManagedRunLoop.start_thread()` now rejects blank thread names and non-boolean
  daemon flags before creating a thread.
- Internal spawned-handle cleanup now inspects whether `dispose()` accepts a
  `timeout=` keyword. Timeout-aware disposers are called once, so their own
  `TypeError` exceptions are preserved instead of being retried as no-argument
  disposal.

### Included current main fixes

- Sensor filtering now rejects invalid key callbacks, detects mapping shape
  changes deterministically, validates frame assembler selectors, and retains
  deterministic key ordering for mixed mapping shapes.
- Component schemas, component record containers, memory chip inputs, memory
  graph/route inputs, primitive enum validation, embedded bulk-credit policy,
  and embedded metadata route validation now fail earlier with stable errors.
- Route retention policies, graph diagnostic node labels, reactive-thread helper
  inputs, and sensor batching helper metadata now reject malformed values before
  graph/runtime side effects occur.
- Read-write step construction now validates malformed inputs before examples or
  runners can build invalid step definitions.
- Example catalog module names, lookup keys, numeric metadata, manifest-order
  diagnostics, malformed lookup coverage, and metadata tests are tighter and
  more deterministic.
- Reactive sequences are materialized as snapshots, latency history defaults no
  longer self-capture, and graph/reactive-thread helper names validate earlier.
- Cargo feature lists and RFC checklist coverage checks remain sorted and
  explicit.

## Compatibility

This is a patch release.

- No public APIs were removed.
- Existing valid waits, joins, disposals, and run-loop thread starts keep their
  behavior.
- Invalid lifecycle timeout values now fail earlier with explicit `ValueError`
  messages.
- Spawned handles whose timeout-aware `dispose()` raises `TypeError` now surface
  that original error after one call.

## Versioning Decision

This release is tagged as 0.1.32 because 0.1.31 is already published, and the
new lifecycle behavior is compatible for valid callers while making invalid
timing and cleanup paths explicit.
