# manyfold 0.1.11

Release date: 2026-05-03

## Summary

manyfold 0.1.11 is a backward-compatible patch release for disk-backed route
memory recovery. It makes corrupt or incomplete `Memory` JSONL files fail with
clear path-and-line diagnostics instead of surfacing low-context JSON, key, or
decode errors during startup.

## What Changed

### Added explicit memory-file corruption diagnostics

`Memory(path)` now validates each non-empty JSONL record while seeding its
duplicate filter. Invalid JSON raises `ValueError` with the memory file path and
line number. JSON values that are not objects, or records missing required
fields such as `payload_b64`, also raise `ValueError` with the same location
context.

Before this release, a truncated or partially written memory file could fail
with arbitrary exceptions such as `JSONDecodeError` or `KeyError`, which made
durable recovery failures harder to diagnose.

### Added regression coverage

The component tests now cover malformed JSONL and missing-field records and
assert that the reported error includes the affected memory file and line.

## Why This Matters

`Memory` is intended for durable route replay and resume. When a persisted file
is damaged, operators need to know which file and record to repair. Clear
corruption errors make recovery deterministic and avoid hiding disk-format
problems behind incidental implementation exceptions.

## Compatibility

This is a patch release.

- No public APIs were removed or renamed.
- No call signatures changed.
- The memory JSONL format is unchanged for valid files.
- Existing valid memory files continue to load normally.
- Invalid memory files now fail with a documented `ValueError` message.

## Versioning Decision

This release is tagged as `0.1.11` rather than `0.2.0` because it hardens an
existing durable recovery path without changing public APIs or valid file
formats.
