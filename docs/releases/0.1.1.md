# manyfold 0.1.1

Release date: 2026-05-02

## Summary

manyfold 0.1.1 is a targeted bug-fix release for the local sensor IO layer. It
fixes a data-loss issue in `DelimitedMessageBuffer(mode="text")` when callers
append UTF-8 data in arbitrary byte chunks and a multibyte character is split
across chunk boundaries.

The bug was subtle because newline-delimited text streams often work in happy
path tests that only exercise ASCII payloads or chunk boundaries that happen to
align with complete characters. In the broken path, the buffer decoded each
incoming byte chunk independently with `errors="ignore"`. If a partial UTF-8
sequence arrived at the end of one chunk, those bytes were discarded before the
next chunk arrived, which permanently removed the character from the resulting
message.

This release changes the text-mode buffer to use a stateful incremental UTF-8
decoder so incomplete character sequences remain pending until enough bytes
arrive to decode them safely.

## What Changed

### Fixed UTF-8 chunk-boundary handling in `DelimitedMessageBuffer`

`DelimitedMessageBuffer(mode="text")` now preserves decoder state across
multiple `append()` calls. Internally, the buffer uses Python's incremental
UTF-8 decoder instead of decoding each byte chunk in isolation.

Before this release:

- `append(b"caf\\xc3")` returned no message and silently dropped the incomplete
  leading byte for `é`
- `append(b"\\xa9\\n")` then produced `"caf"` instead of `"café"`

After this release:

- `append(b"caf\\xc3")` still returns no completed message
- `append(b"\\xa9\\n")` now correctly produces `"café"`

### Reset decoder state on `clear()`

Calling `DelimitedMessageBuffer.clear()` now resets the internal incremental
decoder in addition to clearing the visible text and byte buffers. This keeps
buffer reuse predictable after partial multibyte input.

### Added regression coverage

The sensor IO test suite now includes a regression test that explicitly splits
the UTF-8 encoding of `é` across two `append()` calls and asserts that the
completed text message is preserved.

## Why This Matters

This bug sits on a boundary between raw device IO and parsed application
messages, which is exactly where silent corruption is most expensive.

Affected usage patterns include:

- serial or UART readers that pass arbitrary chunks into a text buffer
- TCP or pipe readers that decode newline-delimited UTF-8 messages incrementally
- sensor bridges that carry human-readable labels, identifiers, or metadata
- JSON-line style device protocols containing non-ASCII text

Without this fix, applications could observe message content that looked valid
but was missing characters. That is worse than a loud decode failure because it
creates ambiguity in debugging and downstream state.

## Compatibility

This is a backward-compatible patch release.

- No public APIs were removed or renamed
- No call signatures changed
- Existing ASCII-only flows continue to behave the same way
- The only behavior change is that split multibyte UTF-8 text is now preserved
  instead of being silently truncated

## Suggested GitHub Release Notes

### Highlights

- Fixed silent UTF-8 character loss in `DelimitedMessageBuffer(mode="text")`
- Added regression coverage for split multibyte input

### Detailed Notes

manyfold `0.1.1` fixes a bug in the text-mode delimited message buffer used by
local sensor and device IO helpers. Previously, when UTF-8 text arrived in
multiple byte chunks and a multibyte character was split across chunk
boundaries, the buffer decoded each chunk independently with ignored errors.
That caused incomplete byte sequences to be dropped, which could silently
truncate characters in completed messages.

The implementation now uses a stateful incremental UTF-8 decoder so partial
characters remain buffered until the remaining bytes arrive. The release also
resets decoder state when the buffer is cleared and adds a regression test that
reproduces the issue with the split-byte encoding of `é`.

This is a backward-compatible patch release and should be a safe upgrade for
existing users.
